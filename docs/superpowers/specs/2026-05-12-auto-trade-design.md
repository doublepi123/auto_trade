# Auto Trade 简化版设计

## 目标

构建一个 Python 自动交易项目，面向长桥 SDK，支持美股和港股单标的区间交易：

- 配置某只股票的最低价 `buy_low` 和最高价 `sell_high`
- 价格触达最低价时，全仓买入
- 价格触达最高价时，全仓卖出
- 可选开启做空：卖出后可反向开空，触达最低价时回补平空
- 前端通过 Web UI 配置参数并查看实时状态
- 运行方式为常驻后台服务，行情通过长桥实时推送驱动策略

## 范围

本期只做以下能力：

- 单个标的的区间交易
- 美股和港股支持
- 实时行情接入
- 下单执行
- 简单风控
- Web UI 配置与状态展示
- Docker 部署
- Server酱通知

## 非目标

- 多标的组合策略
- 高频交易
- 复杂择时指标
- 回测系统
- 量化研究平台
- 代客理财或公开策略分发

## 总体架构

采用前后端分离：

- `backend`：FastAPI 服务，负责配置、策略状态、行情订阅、下单、通知、风控和持久化
- `frontend`：Vue 3 + Vite + Element Plus，负责配置表单、状态面板、订单/成交历史
- `longbridge`：作为券商 SDK，提供行情和交易能力
- `sqlite`：保存策略配置、运行状态、订单、成交、风控事件

数据流：

`Longbridge WebSocket 行情 -> 策略引擎 -> 风控判断 -> 订单执行 -> 数据库存档 -> Web UI 展示 -> Server酱通知`

## 关键组件

### 1. 行情适配器

职责：

- 订阅单个标的的实时行情
- 将长桥 SDK 的行情事件转换成统一内部价格事件
- 处理断线重连和订阅恢复

### 2. 策略引擎

职责：

- 维护当前状态：`flat`、`long`、`short`
- 根据最新价格和当前仓位决定是否触发交易
- 保证同一价格事件只触发一次动作
- 处理买入、卖出、做空和平仓的状态转换

状态机：

- `flat` -> `long`：价格触达 `buy_low`
- `long` -> `flat`：价格触达 `sell_high`
- `flat` -> `short`：开启做空且价格触达 `sell_high`
- `short` -> `flat`：价格触达 `buy_low`

### 3. 风控模块

职责：

- 单日最大亏损控制
- 连续亏损次数限制
- 交易暂停与手动恢复
- 紧急停止开关

风控优先级高于策略信号。只要风控拒绝，策略不能继续发单。

### 4. 交易执行器

职责：

- 根据当前账户可用资金/持仓计算全仓数量
- 生成限价单或市价单的执行请求
- 调用长桥交易接口提交订单
- 轮询或订阅订单状态，记录成交结果

默认以限价单为主，避免滑点失控。若用户启用“立即成交”模式，再由配置决定是否放宽为更激进的下单方式。

### 5. Web UI

职责：

- 配置股票代码、市场、最低价、最高价、是否做空、风控阈值、通知地址
- 展示实时价格、当前状态、持仓、订单、成交、风控事件
- 提供启动、暂停、恢复、kill switch 控制

### 6. 通知模块

职责：

- 下单成功、拒单、成交、风控触发时推送到 Server酱

## 配置模型

核心配置项：

- `symbol`：标的代码
- `market`：`US` 或 `HK`
- `buy_low`：最低价
- `sell_high`：最高价
- `short_selling`：是否启用做空
- `max_daily_loss`：单日最大亏损
- `max_consecutive_losses`：连续亏损上限
- `sct_key`：Server酱发送密钥

配置由 Web UI 写入后端持久化存储，后端重启后仍可恢复。

## 交易规则

### 做多

- 若当前为空仓，且最新价格 `<= buy_low`，则按可用资金全仓买入
- 若当前持有多头，且最新价格 `>= sell_high`，则全仓卖出

### 做空

- 若启用做空，且当前为空仓，且最新价格 `>= sell_high`，则开空
- 若当前持有空头，且最新价格 `<= buy_low`，则买入回补

### 触发去重

同一方向的重复触发需要抑制，避免行情在阈值附近抖动导致连续下单。实现上使用：

- 当前仓位状态
- 最近一次触发价格
- 最近一次触发时间窗口

## 数据持久化

SQLite 保存以下信息：

- 策略配置
- 行情快照的最近值
- 当前运行状态
- 订单记录
- 成交记录
- 风控事件
- 通知发送记录

这样 Web UI、重启恢复和审计都可以依赖同一份本地状态。

## API 设计

后端提供以下接口：

- `GET /api/strategy`：读取当前策略配置
- `PUT /api/strategy`：更新策略配置
- `GET /api/status`：读取实时状态、仓位、最新行情
- `GET /api/orders`：查询订单
- `GET /api/fills`：查询成交
- `POST /api/control/pause`：暂停交易
- `POST /api/control/resume`：恢复交易
- `POST /api/control/kill-switch`：紧急停止

前端通过 WebSocket 或轮询获取实时状态更新。

## 错误处理

- 行情断连：自动重连，恢复后继续评估策略
- 下单失败：记录错误、通知用户、暂停重复触发直到人工确认
- 风控触发：立即停止交易并标记原因
- 配置非法：前端和后端双重校验，禁止保存

## 部署

- 使用 Docker Compose 启动 backend 和 frontend
- 环境变量用于长桥凭证和通知密钥
- SQLite 文件、日志和运行状态挂载到宿主机卷

## 测试策略

- 单元测试覆盖策略状态机
- 单元测试覆盖风控判断
- 单元测试覆盖全仓数量计算
- 单元测试覆盖配置校验
- 集成测试覆盖后端 API
- 集成测试覆盖行情事件驱动下单的主流程

## 实施原则

- 先保证模拟盘可跑通，再接实盘
- 默认限价单，避免把策略变成无保护的市价单机器人
- 策略和风控分离，风控永远优先
- 只做单标的，避免一开始就引入组合管理复杂度
